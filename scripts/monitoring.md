Анатомия bash-скрипта: что происходит после нажатия Enter
Написать #!/bin/bash и пару команд может каждый. Но понимать, что именно творится под капотом, когда этот скрипт запускается — это и есть грань между копипастером и инженером. Давайте разложим процесс на слои: от чтения файла до выполнения последней команды.

1. Шебанг и рождение процесса
Когда вы запускаете ./script.sh, ядро смотрит на первые байты файла. Если находит магическую последовательность #! (шебанг), то читает путь, указанный следом — /bin/bash. Именно этот интерпретатор и запускается как новый дочерний процесс. Если шебанг опущен, система запустит ваш текущий shell, что может привести к сюрпризам (sh вместо bash). Так что #!/bin/bash — не просто ритуал, а способ явно указать «мозг», который будет думать над скриптом.

2. Чтение и разбиение на команды
Bash не выполняет скрипт сразу целиком. Он читает его построчно (точнее, считывает «полную команду» — до разделителя вроде ;, &, &&, ||, новой строки или конца файла). Каждая полученная строка называется списком команд.

Прежде чем хоть что-то выполнить, bash производит лексический анализ, разбивая строку на слова и операторы. Здесь вступают в силу правила quoting (кавычек) и экранирования:

Пробелы и табуляции разделяют слова.

Кавычки (" ", ' ') защищают пробелы и спецсимволы.

Обратный слеш \ отключает специальное значение следующего символа.

Именно на этом этапе bash понимает, что echo "Hello World" — это одно слово-аргумент, а не два.

3. Подстановки: магия превращений
Как только строка разбита на слова, начинается самое интересное — последовательность подстановок (substitutions). Порядок строго фиксирован, и его полезно знать:

Раскрытие фигурных скобок (Brace Expansion).
{1..3} превращается в 1 2 3. Происходит до всего остального.

Замена тильды (Tilde Expansion).
~ становится домашней директорией.

Подстановка переменных и параметров (Parameter Expansion).
$HOME, ${VAR:-default} — самое часто используемое. Здесь же обрабатываются специальные переменные $?, $$ и т.д.

Подстановка команд (Command Substitution).
$(команда) или старый вариант `команда`. Bash запускает дочерний процесс, захватывает его stdout и вставляет результат на место.

Арифметические вычисления (Arithmetic Expansion).
$((2+2)) превращается в 4.

Разделение слов (Word Splitting).
То, что получилось после всех подстановок, повторно разбивается по пробелам, но только если результат не был в двойных кавычках. Классическая ошибка: files="*.txt"; ls $files — переменная без кавычек разобьётся и раскроет маску. А ls "$files" — нет.

Раскрытие глобов (Pathname Expansion).
*.txt превращается в список файлов, удовлетворяющих маске.

Только после этого многослойного пирога bash получает окончательный список аргументов для команды.

4. Исполнение: простые команды и конвейеры
Теперь у нас есть команда и аргументы. Bash смотрит, является ли имя команды встроенной (builtin), функцией, алиасом или внешней программой. Приоритет: алиасы, затем функции, затем builtin, и только потом поиск в $PATH. Если нужна внешняя программа — fork() и exec().

Но чаще команды объединяются в конвейеры (pipes) и логические цепочки:

cmd1 | cmd2 — stdout первого соединяется с stdin второго, оба процесса работают параллельно.

cmd1 && cmd2 — второе выполнится, только если первое завершилось с кодом 0.

cmd1 || cmd2 — второе выполнится, если первое вернуло не ноль.

Весь этот блок команд образует список, и у него будет общий код возврата ($?), который запоминается на будущее.

5. Управляющие конструкции: скелет логики
Ключевые слова if, for, while, case — это встроенные механизмы, которые управляют потоком выполнения, опираясь на коды возврата команд. Важно: if проверяет не булево выражение, а код выхода команды.

bash
if grep -q "error" log.txt; then
    echo "Нашли ошибку"
fi
Здесь grep возвращает 0 (успех), если строка найдена, и if заходит в then-ветку. Никакой «истины» и «лжи» в классическом понимании — только статусы выполнения.

Цикл for итерируется по списку слов, сформированному с учетом всех подстановок из шага 3. while снова и снова выполняет список команд, пока код возврата нулевой.

6. Роль exit code и ловушки ошибок
Каждая команда (даже простое присваивание переменной) устанавливает $?. Скрипт обязан возвращать внятный финальный код через exit N. По умолчанию это код последней выполненной команды.

Опытные скрипты включают «защиту от дурака»:

set -e — завершить скрипт, если любая команда вернула не ноль (с оговорками, связанными с конвейерами и условиями).

set -u — использовать неинициализированную переменную считается ошибкой.

set -o pipefail — кодом возврата конвейера станет первый ненулевой код (а не только последней команды).

Также часто ставят ловушку trap 'cleanup; exit 1' ERR для запуска функции при ошибке.

Заключение: почему это знание спасает
Понимание этой многослойной машины превращает написание скриптов из магии в инженерию. Вы перестаёте гадать, почему не сработало условие в if [ $var == "yes" ] (потому что без кавычек переменная могла разбиться на слова, и test получил не три аргумента, а пять). Вы осознаёте, что $(cat file | grep error) — это избыточно созданный процесс, и лучше прочитать файл напрямую.

Bash — это не просто набор команд, это интерпретатор со своей дисциплиной. Чем лучше вы понимаете порядок, в котором он «думает», тем меньше сюрпризов будет на продакшене.
